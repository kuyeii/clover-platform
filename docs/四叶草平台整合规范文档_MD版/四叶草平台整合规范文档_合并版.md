## 0. 已确认约束

| 确认项 | 结论 |
| --- | --- |
| PostgreSQL 版本 | 开发、测试、生产均统一使用 PostgreSQL 18。 |
| 认证体系 | 第一阶段复用现有 Portal 登录与 session 机制，平台稳定后再升级认证体系。 |

## 1. 总体目标

本行动方案用于指导现有五个独立项目逐步整合为统一平台。总体原则是：先统一入口和运行方式，再统一数据库和后端能力，最后统一前端路由和公共组件。整个过程避免一次性大重构，确保每个阶段完成后系统都可运行、可回退、可验证。

| 目标 | 落地方式 |
| --- | --- |
| 统一维护 | 采用 monorepo 管理代码，保留模块边界，统一脚本、配置、依赖基线和部署方式。 |
| 统一数据库 | 使用 PostgreSQL 18，单库多 schema，core 存公共能力，各业务 schema 存模块数据。 |
| 统一后端 | 中期收敛为一个 FastAPI 主应用，业务通过 router/module接入。 |
| 统一前端 | 第一阶段保留 iframe，第三阶段逐步迁移为统一 React 路由。 |
| 统一配置 | 配置拆分为 default.yaml、env 文件、config.local.yaml、workflows.yaml，支持 dev/test/prod。 |

## 2. 三阶段主线

| 阶段 | 核心任务 | 验收标准 |
| --- | --- | --- |
| 阶段一：稳态收拢 | 建立 monorepo；保留 iframe；增加统一启动器和端口发现；统一配置入口；PostgreSQL 18 环境准备。 | 5 个项目可从 Portal 进入；端口占用时自动分配；配置不再硬编码；每个模块原功能不退化。 |
| 阶段二：后端和数据库统一 | Portal、RAG、竞对、标书、合同审查 | 不再产生新的 SQLite；所有业务通过 |

| 阶段 | 核心任务 | 验收标准 |
| --- | --- | --- |
|  | 逐步切 PostgreSQL；竞对后端改 FastAPI；统一 Dify client、文件元数据、任务表、日志。 | 统一 API 网关或主 FastAPI 暴露；核心流程可回归。 |
| 阶段三：前端统一路由 | 逐步去 iframe；抽公共 UI、api- client、hooks；模块页面迁入统一前端壳。 | 登录态、菜单、权限、历史记录、错误提示统一；模块切换无需 iframe。 |

## 3. 推荐执行顺序

- 第 1 步：建立 clover-platform monorepo，把 5 个项目放入 legacy/ 或 modules/ 下，保持原样可启
动。

- 第 2 步：新增 scripts/dev.py，负责端口探测、启动子进程、生成 runtime/ports.json。
- 第 3 步：改 Portal 的 apps.config.ts，使应用地址从运行时接口读取，避免端口写死。
- 第 4 步：准备 PostgreSQL 18、Alembic、SQLAlchemy 2.x、公共 db session。
- 第 5 步：优先迁 Portal 数据库，因为用户、权限、应用占用是平台核心。
- 第 6 步：迁竞对分析并改 FastAPI，业务复杂度低，适合作为后端统一样板。
- 第 7 步：迁 RAG、标书生成，最后迁合同审查。合同审查涉及 DOCX 定位和导出，放最后最稳。
- 第 8 步：统一 Dify client，所有模块工作流调用都走公共封装。
- 第 9 步：建立 packages/ui、packages/api-client，逐步替代各项目重复组件。
- 第 10 步：去 iframe，按模块逐个迁移页面到统一 React 路由。
## 4. 风险控制和回退策略

| 风险 | 控制方式 | 回退策略 |
| --- | --- | --- |
| 数据库迁移引起业务不可用 | 每次只迁一个模块；保留旧 SQLite 分支；引入基础集成测试。 | 切回模块旧启动方式和旧 SQLite 配置。 |
| 统一后端影响长任务 | 先保持原 API 语义；统一 jobs 表但不立刻引入任务队列。 | 模块 router 独立禁用，临时转发到旧服务。 |
| 前端去 iframe 影响页面样式 | 最后阶段做；先抽公共组件再迁页面。 | 保留 iframe 路由作为 fallback。 |
| Dify workflow key 混乱 | 统一 workflows.yaml；每环境独立变量。 | 回退到模块旧 .env。 |

## 1. 架构定位

平台采用“模块化单体优先、微服务后置”的架构策略。现阶段不拆微服务，不引入复杂服务治理，而是通过统一仓库、统一 FastAPI 后端、统一 PostgreSQL 18、统一配置体系和统一前端壳，降低维护成本。

| 确认项 | 结论 |
| --- | --- |
| 架构层 | 设计结论 |
| 代码组织 | monorepo，modules/ 按业务模块隔离，packages/ 存公共库。 |
| 后端形态 | 第一阶段保留现有服务，第二阶段收敛为一个 FastAPI 主应用。 |
| 前端形态 | 第一阶段保留 Portal iframe，第三阶段迁移为统一 React 路由。 |
| 数据库 | PostgreSQL 18 单库多 schema；core 公共 schema + 业务 schema。 |
| 文件存储 | 本地 volume 优先，core.files 记录元数据，后续可切 MinIO。 |
| 认证 | 复用现有 Portal session，稳定后升级 access token +refresh token。 |

## 2. 目标架构图（文本版）

用户浏览器 -> apps/web 统一前端入口 -> /apps/contract-review -> /apps/bid-generator -> /apps/rag -> /apps/competitor-analysis -> /admin/users -> apps/api 统一 FastAPI 后端 -> core router: auth/users/permissions/files/jobs/settings -> contract_review router -> bid_generator router -> rag router -> competitor_analysis router -> PostgreSQL 18: clover_platform -> core schema -> contract_review schema -> bid_generator schema -> rag schema -> competitor_analysis schema -> 外部能力 -> Dify workflows -> 本地文件 volume，后续可替换 MinIO

## 3. 目录结构建议

clover-platform/ apps/ web/ # 统一前端入口 api/ # 统一 FastAPI 后端入口 modules/ portal/ contract_review/ bid_generator/ rag/ competitor_analysis/ packages/ py_common/ # config/db/auth/dify/storage/jobs/logging ui/ # 前端公共组件 api_client/ # 前端请求封装 shared_types/ # 类型定义 config/ default.yaml apps.yaml workflows.yaml config.local.yaml # 不提交 scripts/ dev.py init_db.py migrate.py docker/ docker-compose.yml

| 当前阶段 | 允许做 | 暂不做 |
| --- | --- | --- |
|  | client、统一文件元数据。 |  |

## 5. 非功能性要求

- 可维护性：业务模块必须边界清晰，不允许跨模块直接读写对方表。
- 可回退性：每个阶段必须可以回退到上一阶段启动方式。
- 可观测性：API 请求、Dify 调用、后台任务、文件导出都要记录统一日志。
- 安全性：认证暂时复用现有机制，但所有 API 必须预留 user context。
- 可扩展性：表结构预留 tenant_id、organization_id、visibility、owner_id。

## 1. 模块划分原则

业务模块按“用户能感知的业务能力”划分，而不是按技术组件划分。Portal、合同审查、标书生成、RAG问答、企业竞对分析均作为一级业务模块。公共能力不得散落在业务模块内，必须沉淀到 core 或 py_common。

- 模块内部可以拥有自己的 service、repository、schema、router、model。
- 模块之间禁止直接 import 对方 service 处理业务，跨模块能力通过 core 服务或事件/任务表协作。
- 用户、权限、文件、任务、审计、配置、Dify 调用属于平台公共能力。
- 短期允许标书生成保留独立逻辑，中期迁入 bid_generator 模块。
## 2. 模块职责表

| 模块 | 职责 | 不负责 |
| --- | --- | --- |
| portal | 统一入口、登录、用户管理、应用权限配置、应用占用状态、反馈入口、菜单配置。 | 不处理合同审查、标书生成、RAG、竞对业务细节。 |
| contract_review | 合同上传、文档解析、风险识别、AI 改写、原文定位、接受/撤销、DOCX 导出。 | 不维护用户权限；不直接管理其他模块文件。 |
| bid_generator | 标书资料管理、脱敏映射、实体/图片注册、大纲生成、正文生成、标书导出。 | 不承担平台统一文件权限；不直接实现用户体系。 |
| rag | 问答会话、联网开关、知识库文档管理、Dify Dataset 代理、会话历史。 | 不负责通用文件中心之外的权限逻辑。 |

| 模块 | 职责 | 不负责 |
| --- | --- | --- |
| competitor_analysis | 企业校验、企业画像、竞对对比、评分、历史记录。 | 不负责 Dify client 通用错误处理。 |
| core | 用户、session、权限、文件元数据、任务、审计、配置、应用注册。 | 不包含具体业务流程。 |

## 3. 公共能力归属

| 公共能力 | 归属 | 说明 |
| --- | --- | --- |
| 认证和当前用户 | core.auth | 第一阶段复用现有 Portal session。 |
| 应用级权限 | core.permissions | 默认普通用户可用五模块，管理员可禁用某用户某模块。 |
| 细粒度权限预留 | core.permissions | 预留 action、resource、scope，不在第一阶段强制落地。 |
| 文件元数据 | core.files | 业务文件实际可在本地 volume，元数据统一入库。 |
| 后台任务 | core.jobs | 先不用 Celery，仅记录状态和进度。 |
| Dify 调用 | py_common.dify | 统一 base_url、timeout、retry、日志、错误处理。 |
| 配置读取 | py_common.config | 多环境配置统一读取。 |

## 4. 数据共享和权限口子

按确认结论，短期所有人的业务数据共享，不做用户隔离。但所有业务表必须预留 owner_user_id、created_by、tenant_id、visibility 或 scope 字段，查询层也要集中封装，避免未来细化权限时大面积改 SQL。推荐字段：

```
tenant_id UUID NULL
```

organization_id UUID NULL

```
owner_user_id UUID NULL
```

created_by UUID NULL

```
visibility TEXT DEFAULT 'shared'
```

access_scope JSONB DEFAULT '{}'短期查询：不过滤 owner_user_id，所有登录用户共享数据。未来查询：按 tenant_id / role / app_permission / resource_permission 过滤。

## 3. 命名规范

| 对象 | 规则 | 示例 |
| --- | --- | --- |
| 表名 | 小写蛇形命名，业务 schema 内不重复带模块前缀。 | contract_review.review_runs |
| 主键 | 统一 id，类型优先 UUID。 | id UUID PRIMARY KEY |
| 外键 | xxx_id。 | owner_user_id |
| 时间字段 | created_at、updated_at、deleted_at。 | created_at timestamptz |
| 布尔字段 | is_ / has_ / enable_ 开头。 | is_active |
| JSON 字段 | 使用 jsonb，不使用 json。 | payload jsonb |

## 4. 通用字段规范

id UUID PRIMARY KEY DEFAULT gen_random_uuid(), tenant_id UUID NULL, organization_id UUID NULL, owner_user_id UUID NULL, created_by UUID NULL, updated_by UUID NULL, created_at timestamptz NOT NULL DEFAULT now(), updated_at timestamptz NOT NULL DEFAULT now(), deleted_at timestamptz NULL, visibility text NOT NULL DEFAULT 'shared', metadata jsonb NOT NULL DEFAULT '{}'::jsonb

## 5. JSONB 使用规范

- 开发阶段允许将 Dify 原始输出、文档解析结果、任务上下文放入 JSONB，降低早期建模成本。
- 需要列表筛选、排序、权限判断、统计的字段应拆成独立列。
- JSONB 字段必须有结构说明，不允许长期成为不可控垃圾桶。
- 常用 JSONB 查询字段需要加 GIN 或表达式索引。
## 6. 迁移规范

- 所有 DDL 变更必须通过 Alembic migration 提交。
- 一个 migration 只做一类变更，禁止混合大量无关表结构修改。
- 开发阶段可 reset 数据库，但 migration 文件仍要保持从空库可完整初始化。
- 禁止业务启动时自动 create_all 替代 migration。

| 本地覆盖 | config/config.local.yaml | 不提交 | 本机路径、调试开关。 |
| 环境变量 | .env / Docker secrets | 不提交 | 数据库、密钥、Dify API key。 |

## 2. 配置优先级

## 4. 后端接入规范

- 每个模块必须暴露 router.py，由 apps/api/main.py include_router。
- 每个模块必须有 service 层和 repository 层，禁止 router 直接写复杂业务。
- 数据库模型必须声明 schema，不允许默认 public。
- Dify 调用必须通过 py_common.dify。
- 文件读写必须通过 py_common.storage，并写入 core.files。
```
app.include_router(contract_review.router, prefix="/api/v1/contract-review")
app.include_router(bid_generator.router, prefix="/api/v1/bid-generator")
app.include_router(rag.router, prefix="/api/v1/rag")
app.include_router(competitor_analysis.router, prefix="/api/v1/competitor-analysis")
```

## 5. 前端接入规范

- 菜单从 /api/v1/core/apps 获取，不再在前端硬编码完整列表。
- 阶段一 iframe 地址来自 runtime config。
- 阶段三模块页面必须接入统一布局、统一错误页、统一权限拦截。
- 业务模块不得自行实现登录页。
## 6. 接入验收清单

| 确认项 | 结论 |
| --- | --- |
| 检查项 | 通过标准 |
| 应用注册 | core.apps 中存在模块记录，菜单可显示。 |
| 权限控制 | 管理员可禁用某用户访问该模块。 |
| 健康检查 | 启动器能判断模块是否可用。 |
| 数据库 | 模块表在对应 schema 下，不使用 SQLite。 |
| 配置 | 模块没有硬编码端口、Dify key、文件路径。 |
| 日志 | 关键操作进入统一日志。 |
