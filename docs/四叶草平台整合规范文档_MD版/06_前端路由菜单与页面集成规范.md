## 0. 已确认约束

| 确认项 | 结论 |
| --- | --- |
| PostgreSQL 版本 | 开发、测试、生产均统一使用 PostgreSQL 18。 |
| 认证体系 | 第一阶段复用现有 Portal 登录与 session 机制，平台稳定后再升级认证体系。 |
| 多租户 | 当前不做复杂多租户，但表结构预留 tenant_id / organization_id 等扩展口子。 |
| 标书生成服务边界 | 短期保留标书生成现有独立逻辑，中期合并为 bid_generator 模块。 |
| 任务队列 | 第一阶段不引入 Celery/RQ，后续按长任务压力再评估。 |
| iframe 策略 | 第一阶段保留 iframe，第二阶段统一后端和数据库，第三阶段逐步去 iframe。 |
| 文件存储 | 暂时使用本地目录或 Docker volume，后续可平滑切换 MinIO。 |
| 权限策略 | 默认普通用户可使用五个模块，管理员可配置某用户不能使用某模块；所有数据短期共享。 |
| 数据隔离 | 暂时不做隔离，但业务表和查询接口预留 user_id、tenant_id、visibility、scope 等口子。 |
| 多环境配置 | 配置体系支持 dev/test/prod 多环境，Dify workflow key 按环境隔离。 |

## 1. 前端集成策略

前端整合分三步走。第一阶段保留 Portal iframe，保证五个项目可用；第二阶段统一 api-client 和运行时配置；第三阶段逐步去 iframe，将业务页面迁移为统一 React 路由。

| 阶段 | 页面形态 | 说明 |
| --- | --- | --- |
| 阶段一 | Portal + iframe | 最小改动，使用 apps 配置加载五个业务系统。 |
| 阶段二 | Portal + runtime config + shared api- client | 后端和数据库统一，但页面仍可 iframe。 |
| 阶段三 | 统一 React routes | 去 iframe，共享菜单、顶部栏、权限、API、主题和组件。 |

## 2. 菜单和路由规范

/apps/bid-generator /apps/contract-review /apps/rag /apps/competitor-analysis /admin/users /admin/settings /feedback

| 字段 | 说明 |
| --- | --- |
| code | 应用唯一编码，例如 contract-review。 |
| name | 显示名称，例如合同审查。 |

| 字段 | 说明 |
| --- | --- |
| routePath | 统一前端路由。 |
| iframeUrl | 阶段一/二使用，阶段三可为空。 |
| icon | 菜单图标。 |
| enabled | 是否启用。 |
| permission | 当前用户是否有权访问。 |

## 5. 权限显示规则

- 普通用户默认看到五个模块。
- 管理员禁用某用户某模块后，该用户菜单中应用置灰或隐藏。
- 如果应用被置灰，点击时展示“当前账号无权访问该模块，请联系管理员”。
- 应用占用状态仍通过 core.app_usage_sessions 展示。
- 未来可扩展到按钮级权限，但第一阶段只做应用级权限。
## 6. 去 iframe 迁移顺序

| 优先级 | 模块 | 原因 |
| --- | --- | --- |
| 1 | 竞对分析 | 页面相对独立，后端改 FastAPI 后迁移成本低。 |
| 2 | RAG 问答 | 页面交互清晰，主要处理 SSE 和会话。 |

| 优先级 | 模块 | 原因 |
| --- | --- | --- |
| 3 | 标书生成 | 流程复杂，但和统一文件/任务体系收益大。 |
| 4 | 合同审查 | DOCX 预览、定位、接受/撤销复杂，放最后更稳。 |
