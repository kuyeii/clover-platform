| 项目 | 内容 |
| --- | --- |
| 版本 | v1.1（基于确认版约束） |
| 日期 | 2026-05-18 |

## 0. 已确认约束

| 确认项 | 结论 |
| --- | --- |
| PostgreSQL 版本 | 开发、测试、生产均统一使用 PostgreSQL 18。 |
| 认证体系 | 第一阶段复用现有 Portal 登录与 session 机制，平台稳定后再升级认证体系。 |
| 多租户 | 当前不做复杂多租户，但表结构预留 tenant_id / organization_id 等扩展口子。 |
| 标书生成服务边界 | 短期保留标书生成现有独立逻辑，中期合并为 bid_generator 模块。 |
| 任务队列 | 第一阶段不引入 Celery/RQ，后续按长任务压力再评估。 |
| iframe 策略 | 第一阶段保留 iframe，第二阶段统一后端和数据库，第三阶段逐步去 iframe。 |
| 文件存储 | 暂时使用本地目录或 Docker volume，后续可平滑切换 MinIO。 |
| 权限策略 | 默认普通用户可使用五个模块，管理员可配置某用户不能使用某模块；所有数据短期共享。 |
| 数据隔离 | 暂时不做隔离，但业务表和查询接口预留 user_id、tenant_id、visibility、scope 等口子。 |
| 多环境配置 | 配置体系支持 dev/test/prod 多环境，Dify workflow key 按环境隔离。 |

## 1. 配置分层

配置采用“默认配置 + 本地覆盖 + 环境变量 + 多环境 workflow 配置”的四层结构。敏感信息只允许放环境变量或部署平台密钥，不提交 git。

| 层级 | 文件/来源 | 是否提交 | 用途 |
| --- | --- | --- | --- |
| 默认配置 | config/default.yaml | 提交 | 非敏感默认值。 |
| 应用注册 | config/apps.yaml | 提交 | 模块注册、路由、端口范围。 |
| 工作流配置 | config/workflows.yaml | 提交 key 名称，不提交密钥值 | Dify workflow 逻辑编码和 env 映射。 |
| 本地覆盖 | config/config.local.yaml | 不提交 | 本机路径、调试开关。 |
| 环境变量 | .env / Docker secrets | 不提交 | 数据库、密钥、Dify API key。 |

## 2. 配置优先级

环境变量 > config.local.yaml > config/{env}.yaml > config/default.yaml配置读取必须通过 py_common.config，不允许业务模块自行读取多个 .env 或硬编码路径。

## 3. 多环境配置示例

app:env: dev base_url: http://localhost:5200 database:url_env: DATABASE_URL storage:backend: local local_base_dir: ./storage modules:contract_review:enabled: true bid_generator:enabled: true security:auth_mode: portal_session future_token_mode: jwt

## 6. 禁止项

- 禁止在前端代码中写死 http://localhost:xxxx 作为业务 API。
- 禁止将 Dify API key、数据库密码提交到 git。
- 禁止每个模块维护互相冲突的配置读取逻辑。
- 禁止生产环境依赖动态端口随机分配。
