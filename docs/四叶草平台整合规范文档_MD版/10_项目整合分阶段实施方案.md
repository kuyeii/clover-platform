# 项目整合分阶段实施方案

> 来源文件：`10_项目整合分阶段实施方案.pdf`

| 项目 | 内容 |
| --- | --- |
| 版本 | v1.1（基于确认版约束） |
| 日期 | 2026-05-18 |
| 适用范围 | Portal、合同审查、标书生成、RAG 问答、企业竞对分析 |
| 关键约束 | PostgreSQL 18；第一阶段保留 iframe；暂用现有认证；数据短期共享；配置多环境化 |

## 0. 已确认约束

| 确认项 | 结论 |
| --- | --- |
| PostgreSQL 版本 | 开发、测试、生产均统一使用 PostgreSQL 18。 |
| 认证体系 | 第一阶段复用现有 Portal 登录与 session 机制，平台稳定后再升级认证体系。 |
| 多租户 | 当前不做复杂多租户，但表结构预留 tenant_id / organization_id 等扩展口子。 |
| 标书生成服务边界 | 短期保留标书生成现有独立逻辑，中期合并为 bid_generator 模块。 |
| 任务队列 | 第一阶段不引入 Celery/RQ，后续按长任务压力再评估。 |
| iframe 策略 | 第一阶段保留 iframe，第二阶段统一后端和数据库，第三阶段逐步去 iframe。 |
| 文件存储 | 暂时使用本地目录或 Docker volume，后续可平滑切换 MinIO。 |
| 权限策略 | 默认普通用户可使用五个模块，管理员可配置某用户不能使用某模块；所有数据短期共享。 |
| 数据隔离 | 暂时不做隔离，但业务表和查询接口预留 user_id、tenant_id、visibility、scope 等口子。 |
| 多环境配置 | 配置体系支持 dev/test/prod 多环境，Dify workflow key 按环境隔离。 |

## 1. 阶段划分总览

| 阶段 | 周期建议 | 核心目标 |
| --- | --- | --- |
| 阶段 0 | 1-2 天 | 冻结当前可运行状态，建立基线和回归清单。 |
| 阶段 1 | 3-5 天 | monorepo、统一启动、动态端口、配置集中、iframe 保留。 |
| 阶段 2 | 5-10 天 | PostgreSQL 18 基础设施、Portal 数据迁移、权限和文件元数据统一。 |
| 阶段 3 | 7-14 天 | 竞对/RAG/标书/合同逐步接入 PostgreSQL 和统一 API。 |
| 阶段 4 | 7-14 天 | 统一 FastAPI 主应用、统一 Dify client、统一 jobs。 |
| 阶段 5 | 持续迭代 | 逐步去 iframe，统一前端路由和公共组件。 |

## 2. 阶段 0：基线冻结

- 为五个项目分别记录启动命令、端口、环境变量、核心流程。
- 建立最小回归清单：登录、进入模块、上传文件、运行 Dify、下载结果、历史记录。
- 保存当前代码分支，例如 integration-baseline。
- 确认历史数据不迁移，可清空 SQLite。
## 3. 阶段 1：运行方式统一

| 确认项 | 结论 |
| --- | --- |
| 任务 | 产出 |
| 建立 clover-platform 仓库结构 | apps、modules、packages、config、scripts、docker。 |
| 加入 scripts/dev.py | 自动探测端口、启动服务、写 runtime/ports.json。 |
| Portal 读取运行时端口 | 避免 iframe URL 写死。 |
| 整理 config/apps.yaml | 模块注册统一。 |
| 整理 .env.example | 列出所有必填环境变量。 |

## 4. 阶段 2：数据库统一基础

- 启动 PostgreSQL 18。
- 创建 core、contract_review、bid_generator、rag、competitor_analysis schema。
- 建立 Alembic。
- 实现 py_common.db。
- 迁 Portal 用户、session、权限、占用状态表。
- 实现 core.files、core.jobs、core.workflow_runs。
## 5. 阶段 3：业务模块逐个迁移

| 顺序 | 模块 | 理由 | 验收 |
| --- | --- | --- | --- |
| 1 | 竞对分析 | 后端最轻，适合作为 SQLite到 PostgreSQL 和 FastAPI改造样板。 | 分析、历史、缓存可用。 |
| 2 | RAG | 流程清晰，主要是会话和知识库代理。 | SSE 问答、文档管理可用。 |
| 3 | 标书生成 | 短期保留独立逻辑，中期并入 bid_generator。 | 项目、映射、生成、导出可用。 |
| 4 | 合同审查 | DOCX 定位和导出最复杂，最后迁移。 | 审查、接受、撤销、导出可用。 |

## 6. 阶段 4：统一后端

- apps/api/main.py 成为唯一后端入口。
- 业务模块以 include_router 接入。
- 所有 Dify 调用改走 py_common.dify。
- 所有文件写入改走 py_common.storage。
- 所有长任务记录 core.jobs。
- 保留旧 API path 的兼容代理，前端逐步切换。
## 7. 阶段 5：统一前端

- 建立 packages/ui 和 packages/api-client。
- 先迁竞对分析，再迁 RAG，再迁标书生成，最后迁合同审查。
- 每迁一个模块，保留 iframe fallback 一个迭代周期。
- 统一菜单、顶部栏、权限拦截、错误页、Loading、文件上传、任务进度。
## 8. 每阶段验收标准

| 阶段 | 必须通过的检查 |
| --- | --- |
| 阶段 1 | 一键启动；端口冲突可自动切换；Portal 可进入五个模块。 |
| 阶段 2 | PostgreSQL 18 可初始化；Portal 不再使用 SQLite；权限配置可用。 |
| 阶段 3 | 每个业务模块迁完后不产生新 SQLite；核心业务流程通过。 |
| 阶段 4 | 只启动统一 API 也能支撑主要业务；Dify 日志统一可查。 |
| 阶段 5 | 模块页面脱离 iframe 后可用；权限、菜单、历史、下载不退化。 |
