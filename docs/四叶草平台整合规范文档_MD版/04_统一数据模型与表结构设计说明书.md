# 统一数据模型与表结构设计说明书

| 项目 | 内容 |
| --- | --- |
| 版本 | v1.1（基于确认版约束） |
| 日期 | 2026-05-18 |
| 适用范围 | Portal、合同审查、标书生成、RAG 问答、企业竞对分析 |

## 0. 已确认约束

| 确认项 | 结论 |
| --- | --- |
| PostgreSQL 版本 | 开发、测试、生产均统一使用 PostgreSQL 18。 |
| 认证体系 | 第一阶段复用现有 Portal 登录与 session 机制，平台稳定后再升级认证体系。 |
| 多租户 | 当前不做复杂多租户，但表结构预留 tenant_id / organization_id 等扩展口子。 |
| 标书生成服务边界 | 短期保留标书生成现有独立逻辑，中期合并为 bid_generator 模块。 |
| 任务队列 | 第一阶段不引入 Celery/RQ，后续按长任务压力再评估。 |
| iframe 策略 | 第一阶段保留 iframe，第二阶段统一后端和数据库，第三阶段逐步去 iframe。 |
| 文件存储 | 暂时使用本地目录或 Docker volume，后续可平滑切换 MinIO。 |
| 权限策略 | 默认普通用户可使用五个模块，管理员可配置某用户不能使用某模块；所有数据短期共享。 |
| 数据隔离 | 暂时不做隔离，但业务表和查询接口预留 user_id、tenant_id、visibility、scope 等口子。 |
| 多环境配置 | 配置体系支持 dev/test/prod 多环境，Dify workflow key 按环境隔离。 |

## 1. 数据模型分层

统一数据模型分为 core 公共模型和业务模块模型。core 负责跨模块共享能力，业务 schema 只保存该模块业务数据。短期数据共享，但模型中保留未来隔离字段。

| 层级 | 典型表 | 说明 |
| --- | --- | --- |
| core 公共层 | users、sessions、apps、app_permissions、files、jobs、audit_logs、workflow_runs | 平台级公共数据。 |
| 业务运行层 | review_runs、projects、conversations、history_records | 业务主流程记录。 |
| 业务明细层 | risk_items、generated_documents、messages、company_profiles | 业务内部明细和缓存。 |
| 扩展层 | settings、metadata jsonb、access_scope jsonb | 为权限细化、多租户、多环境扩展预留。 |

## 2. core 表结构建议

| 表 | 关键字段 | 用途 |
| --- | --- | --- |
| core.users | id, username, display_name, password_hash, role, is_active, metadata | 用户基础表，继承现有 Portal 用户逻辑。 |
| core.sessions | id, user_id, token_hash, expires_at, revoked_at | 第一阶段复用 session。 |

| 表 | 关键字段 | 用途 |
| --- | --- | --- |
| core.apps | id, code, name, route_path, iframe_url, enabled, config | 应用注册表。 |
| core.user_app_permissions | user_id, app_code, allowed, reason | 管理员配置某用户是否可用某模块。 |
| core.app_usage_sessions | app_code, user_id, entered_at, last_seen_at, status | 应用占用状态。 |
| core.files | id, module_code, storage_backend, storage_key, original_name, mime_type, size_bytes | 统一文件元数据。 |
| core.jobs | id, module_code, job_type, status, progress, payload, result, error | 统一长任务状态。 |
| core.workflow_runs | id, module_code, workflow_code, provider, status, request_payload, response_payload | Dify 调用记录。 |
| core.audit_logs | actor_user_id, action, resource_type, resource_id, detail | 审计日志。 |

## 4. 示例 DDL 片段

```
CREATE TABLE core.apps (
id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
code text NOT NULL UNIQUE,
name text NOT NULL,
route_path text NOT NULL,
iframe_url text NULL,
enabled boolean NOT NULL DEFAULT true,
config jsonb NOT NULL DEFAULT '{}'::jsonb,
created_at timestamptz NOT NULL DEFAULT now(),
updated_at timestamptz NOT NULL DEFAULT now()
);
```

```
CREATE TABLE core.user_app_permissions (
id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
user_id uuid NOT NULL REFERENCES core.users(id),
app_code text NOT NULL REFERENCES core.apps(code),
allowed boolean NOT NULL DEFAULT true,
reason text NULL,
created_at timestamptz NOT NULL DEFAULT now(),
UNIQUE(user_id, app_code)
);
```

```
CREATE TABLE core.files (
id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
tenant_id uuid NULL,
module_code text NOT NULL,
owner_user_id uuid NULL,
storage_backend text NOT NULL DEFAULT 'local',
storage_key text NOT NULL,
original_name text NOT NULL,
mime_type text NULL,
size_bytes bigint NULL,
visibility text NOT NULL DEFAULT 'shared',
metadata jsonb NOT NULL DEFAULT '{}'::jsonb,
created_at timestamptz NOT NULL DEFAULT now()
);
```

## 5. 数据共享策略

按确认结论，短期业务数据不做个人隔离，所有登录用户共享可见；管理员拥有全部配置能力。为了未来权限细化，所有新增查询必须通过 repository/service 层，禁止在前端或 router 中手写散落 SQL。
