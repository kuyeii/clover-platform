## 0. 已确认约束

| 确认项 | 结论 |
| --- | --- |
| PostgreSQL 版本 | 开发、测试、生产均统一使用 PostgreSQL 18。 |
| 认证体系 | 第一阶段复用现有 Portal 登录与 session 机制，平台稳定后再升级认证体系。 |
| 多租户 | 当前不做复杂多租户，但表结构预留 tenant_id / organization_id 等扩展口子。 |
| 标书生成服务边界 | 短期保留标书生成现有独立逻辑，中期合并为 bid_generator 模块。 |
| 任务队列 | 第一阶段不引入 Celery/RQ，后续按长任务压力再评估。 |
| iframe 策略 | 第一阶段保留 iframe，第二阶段统一后端和数据库，第三阶段逐步去 iframe。 |
| 文件存储 | 暂时使用本地目录或 Docker volume，后续可平滑切换 MinIO。 |
| 权限策略 | 默认普通用户可使用五个模块，管理员可配置某用户不能使用某模块；所有数据短期共享。 |
| 数据隔离 | 暂时不做隔离，但业务表和查询接口预留 user_id、tenant_id、visibility、scope 等口子。 |
| 多环境配置 | 配置体系支持 dev/test/prod 多环境，Dify workflow key 按环境隔离。 |

## 1. 模块接入目标

任何新模块或旧模块接入平台时，都必须通过统一注册表声明应用信息、权限编码、路由、API 前缀、文件目录、工作流配置和健康检查。这样 Portal、后端、部署脚本和权限系统才能自动识别模块。

## 2. 模块注册字段

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| code | 是 | 应用唯一编码，禁止变更。 |
| name | 是 | 用户可见名称。 |
| description | 是 | 应用说明。 |
| route_path | 是 | 统一前端路由。 |
| api_prefix | 是 | 统一 API 前缀。 |
| iframe_url | 阶段一必填 | iframe 过渡地址，第三阶段可为空。 |
| permission_default | 是 | 默认是否允许普通用户访问，当前默认为 true。 |
| health_check | 是 | 健康检查接口。 |
| storage_namespace | 是 | 文件存储命名空间。 |
| workflow_refs | 否 | Dify workflow 引用。 |

## 4. 后端接入规范

- 每个模块必须暴露 router.py，由 apps/api/main.py include_router。
- 每个模块必须有 service 层和 repository 层，禁止 router 直接写复杂业务。
- 数据库模型必须声明 schema，不允许默认 public。
- Dify 调用必须通过 py_common.dify。
- 文件读写必须通过 py_common.storage，并写入 core.files。
```
app.include_router(contract_review.router, prefix="/api/v1/contract-review")
app.include_router(bid_generator.router, prefix="/api/v1/bid-generator")
app.include_router(rag.router, prefix="/api/v1/rag")
app.include_router(competitor_analysis.router, prefix="/api/v1/competitor-analysis")
```

## 5. 前端接入规范

- 菜单从 /api/v1/core/apps 获取，不再在前端硬编码完整列表。
- 阶段一 iframe 地址来自 runtime config。
- 阶段三模块页面必须接入统一布局、统一错误页、统一权限拦截。
- 业务模块不得自行实现登录页。
## 6. 接入验收清单

| 确认项 | 结论 |
| --- | --- |
| 检查项 | 通过标准 |
| 应用注册 | core.apps 中存在模块记录，菜单可显示。 |
| 权限控制 | 管理员可禁用某用户访问该模块。 |
| 健康检查 | 启动器能判断模块是否可用。 |
| 数据库 | 模块表在对应 schema 下，不使用 SQLite。 |
| 配置 | 模块没有硬编码端口、Dify key、文件路径。 |
| 日志 | 关键操作进入统一日志。 |
