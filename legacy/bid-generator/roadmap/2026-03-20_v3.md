- [x] 无 task_id 的 generating 块：不再显示"连接中断，请重试"（用户不可控错误），改为静默 idle

  - 后端已 **error** → 静默 idle（后端自身错误，非用户可控）
  - 后端仍 **running** → 进入标准 `resumeContentTask` 轮询
  - **网络异常** → fallback 进入标准轮询

## 竞态约束综合评估

| 场景 | 处理策略 | 原理 |
|------|---------|------|
| 刷新前任务刚完成，未及写 localStorage | 后端 done 优先 | fetchData 时主动 GET /status 校验 |
| 刷新前任务未完成 | 继续轮询 | resumeContentTask 10s 轮询 |
| 后端重启（task_id 过期，404）| 静默 idle | onExpired 回调 |
| 用户主动取消 | cancelled 状态 | 前后端同时取消 |
| localStorage generating 但无 task_id | 静默 idle | 不报错 |
| Dify streaming 任务 | Stop API 真实中断 | task_routes._stop_dify_workflow |

> **时间戳方案**：已评估，不采用。原因：中断场景下（网络断开、后端重启），前端任务无法将 updatedAt 写回后端，导致后端的时间戳永远落后于 localStorage。"最终状态优先"（done > generating，idle > error）比时间戳比较更可靠。
