# 入口访问方案说明

## 当前阶段方案
当前阶段采用“门户壳层 + iframe 嵌入”方式访问四个业务模块。门户保留顶部导航、总览页、知识库页和设置页；业务模块继续以独立前端进程运行，在门户下方容器内嵌展示。

## 为什么不直接合并四个业务模块
- 四个模块当前仍处于独立演进状态，技术栈和目录结构并未统一。
- 直接合并会放大改造范围，影响现有可运行性和回滚能力。
- 当前治理原则要求先隔离、再规范、再统一入口，避免大爆炸式整合。

## 当前方案边界
- `portal-launchpad` 只读取模块入口配置，不 import 业务模块源码。
- 业务模块内部路由仍由各自前端自行维护，Portal 只负责切换应用壳层路由。
- 若业务模块后续需要把内部路由态同步给 Portal，需要新增 `postMessage` 通信协议。

## 后续演进方向
- 可演进为父子页面路由同步。
- 可演进为微前端模块接入。
- 可演进为统一路由和统一鉴权。
- 可在平台服务成熟后接入统一网关和统一任务中心。

## 与 02-app-modules 的关系
- `02-app-modules` 是正式业务模块落点。
- `portal-launchpad` 只读取模块入口配置，不 import 业务模块源码。
- 模块完成规范化后，再逐步接入门户统一展示。

## 与 05-platform-services 的关系
- 当前门户不依赖真实平台后端。
- 后续可接入统一登录、用户中心、知识库中心、任务中心和审计日志服务。
- 当前页面中的工作台、知识库、设置页均为平台化能力预留。
